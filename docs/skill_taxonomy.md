
# Построение таксономии навыков с бюджетом solo‑разработчика

Вы можете запустить производственную платформу IT‑навыков для данных масштаба HH.ru примерно за **$5–10 в месяц на LLM**, если объединить:
- **ESCO** как лицензированную основу таксономии,
- структурированные поля **HH.ru** как бесплатный источник информации о сеньорности,
- **Gemini 2.5 Flash‑Lite** (или **GPT‑5 nano**) в batch‑режиме для извлечения навыков.

Главная идея: почти каждая сложная задача здесь — определение уровня специалиста, извлечение навыков и построение иерархии — уже имеет **бесплатный структурированный сигнал** либо в исходных данных, либо в открытых датасетах ЕС и США.  
Поэтому **LLM должны обрабатывать только 15–30% случаев**, которые не решаются правилами и словарями.

Вторая важная идея: для российского рынка нужен **двухслойный дизайн словаря навыков**:
- канонический словарь на **английском**
- таблица **кириллических алиасов**
- обработка **Unicode‑гомоглифов**

Причина: «С#» (кириллическая С) и «C#» (латинская C) визуально одинаковы для пользователя, но являются разными символами Unicode и создают дубликаты навыков в наивных пайплайнах.

# Основа таксономии: ESCO + Lightcast

Ни одна открытая таксономия не подходит идеально.

### ESCO v1.2.1
- ~13 939 навыков  
- 28 языков  
- бесплатная лицензия EU Decision 2011/833/EU

Это единственный стандарт, который:
- полностью скачивается
- содержит богатую сеть отношений
- разрешён для коммерческого использования с указанием источника

Полезные файлы:
- `skillsHierarchy`
- `broaderRelationsSkillPillar`
- `skillSkillRelations`
- `occupationSkillRelations`

Последний особенно важен — он содержит флаг **essential / optional** для пары *(профессия, навык)*, что позволяет выявлять **пререквизиты навыков**.

Недостатки:
- слабая детализация в IT  
- есть «C#» и «relational databases», но нет «EF Core» или «Npgsql»  
- отсутствует **русский перевод**

# Нормализация: проблема кириллических гомоглифов

Обычный fuzzy‑matching решает простые случаи.  
Но в русскоязычном IT‑корпусе есть специфическая проблема — **Unicode confusables**.

Например:
- «С#» (кириллическая С)
- «C#» (латинская C)

Это разные символы Unicode и поэтому строки считаются разными.

Решение — **UTS #39 skeleton mapping**.

Каждый символ заменяется каноническим представителем класса визуально похожих символов.

Так:
- «С#»
- «c#»
- «с#»

дают одинаковый внутренний `match_key`.

В C# это можно реализовать через пакет **ICU4N**:

`SpoofChecker.GetSkeleton()`

Важно: **никогда не показывайте skeleton пользователю**, это только внутренний ключ эквивалентности.

# Пайплайн нормализации

1. NFKC нормализация  
2. case‑fold (через `CultureInfo.InvariantCulture`)  
3. confusable skeleton  
4. удаление zero‑width символов  
5. нормализация пробелов

Хранить нужно:

- `display_name` — исходная строка
- `match_key` — нормализованная строка

Дополнительно поддерживайте таблицу **транслитерационных алиасов**:

| Русский | Канонический |
|-------|------|
| Си‑шарп | c# |
| Питон | python |
| Яваскрипт | javascript |

# Токенизация технических терминов

Перед разделением текста необходимо защищать термины:

- C++
- C#
- .NET
- Node.js
- F#
- ASP.NET
- gRPC

Практичный подход — **двухпроходная токенизация**:

1. заменить термины на плейсхолдеры (`__CPP__`, `__DOTNET__`)
2. выполнить обычную токенизацию
3. вернуть оригинальные значения

Также полезно хранить **collapsed‑ключ**:

```
c# → csharp
.net → dotnet
```

Это позволяет автоматически сопоставлять разные написания.

# Метрики похожести навыков

Оптимальный стек:

1. **trigram blocking**
2. **Damerau‑Levenshtein**
3. **семантические embeddings**

Фонетические алгоритмы вроде Soundex или Metaphone **не подходят для IT‑терминов**.

В PostgreSQL удобно использовать:

- `pg_trgm`
- GIN‑индекс по `match_key`

# Семантический слой (embeddings)

Для аббревиатур вроде:

- JS ↔ JavaScript
- K8s ↔ Kubernetes

нужен семантический слой.

Рекомендуемые модели:

- `intfloat/multilingual-e5-base`
- `multilingual-e5-small`
- `cointegrated/LaBSE-en-ru`

Хранение:
- `pgvector`
- индекс **HNSW**

Типичные пороги сходства:

| Косинус | Действие |
|------|------|
| ≥0.92 | авто‑объединение |
| 0.85–0.92 | ручная проверка |
| <0.85 | отклонить |

# Граф навыков

Все рекомендации строятся на **графе навыков**, полученном из co‑occurrence.

Основная метрика — **Normalized PMI (NPMI)**.

Формула:

```
NPMI = PMI / -log P(A,B)
```

Диапазон:

```
[-1, +1]
```

Фильтр рёбер:

```
c_AB ≥ 20
NPMI ≥ 0.2
G² ≥ 10.83 (p < 0.001)
```

# Заключение

Этот проект реализуем одним разработчиком, потому что **самые сложные задачи уже имеют бесплатные структурированные сигналы**:

- сеньорность — из поля experience
- иерархия — из ESCO
- связи навыков — из NPMI
- семантика — из embeddings

LLM обрабатывают только **небольшой хвост сложных случаев**.

Поэтому итоговая стоимость системы — всего **$3–5 в месяц**, а масштабирование практически ничего не стоит.
